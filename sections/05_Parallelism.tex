\section{Parallelism}

\subsection{Levels of Parallelism}
A single processor at ever-higher clock hits limits (complexity, power), so efficiency comes from parallelism:
\begin{outline}
  \1 \textbf{ILP} (instruction-level): overlap instruction stages (pipelining, superscalar, VLIW) --- gains at the same clock.
  \1 \textbf{TLP} (thread-level): several concurrent threads (own stack/registers/PC) when one program lacks independent instructions.
  \1 \textbf{DLP} (data-level) \textrightarrow{} \textbf{SIMD}: one instruction for many data elements. 
  \textbf{MIMD} \textrightarrow{} multiple instructions for multiple data (more efficient than SIMD) 
\end{outline}

\subsection{Pipelining}
% GOAL V11 • You can explain how pipelining and its variations like superscalar or VLIW architectures work.

Split a task into \textbf{stages} processed in overlap (assembly line). 
\begin{itemize}
    \item \textbf{Fetch:} Load instruction from program memory
    \item \textbf{Decode:} decode instruction, load address and operand from registers
    \item \textbf{Execute:} Execute task of instruction or calculate address for following memory access
    \item \textbf{Memory Access:} Access memory (inkl. Cache) if used in instruction, else it is skipped
    \item \textbf{Write Back:} Write back to registers 
\end{itemize}
\textbf{Pipeline registers} hold data between stages (transparent to the user). Ideal: one finished instruction \emph{per cycle}. 
\[ \text{Speedup} = \frac{\text{CPI unpipelined}}{1 + \text{stall cycles per instruction}} = \frac{\text{Pipeline depth}}{1 + \text{stall cycles per instruction}} \]
Limits: bus access times, pipeline-register delay, hazards.

\subsection{Pipeline Hazards}
% GOAL V11 • You can assess challenges associated with pipelines like hazards and describe possible solutions.
Three types: \textbf{structural} (resource conflicts), \textbf{data}, \textbf{control}.
\begin{outline}
  \1 \textbf{Data:} \emph{RAW} (read-after-write, the true/most common dependency --- solved by \textbf{data forwarding} or compiler scheduling), \emph{WAW} (write order, compiler), \emph{WAR} (anti-dependency, relevant out-of-order).
  \1 \textbf{Control:} branches (\(\sim\)20\% of ops, \(\sim\)66\% of conditional ones taken). 
  Solutions: \emph{look-ahead resolution} (compute target in parallel), \emph{delay slot}, \emph{branch prediction}.
\end{outline}
Interrupts/exceptions are handled by saving state \textrightarrow{} ISR \textrightarrow{} resume (extra HW, watch delays).

\subsection{Superscalar \& VLIW}
\textbf{Superscalar:} duplicated functional units issue $>1$ instruction/cycle ($\text{CPI}<1$); 
abandons strict Von-Neumann order \textrightarrow{} \textbf{out-of-order} execution, shifting focus from \emph{instruction} to 
\emph{result} sequentiality (hazards more severe, solve at compile time). \textbf{Superpipeline:} finer, shorter stages at higher clock (mixes exist). 
\textbf{VLIW:} several operations packed in one long instruction word, scheduled by the \textbf{compiler} (avoids hazards, simplifies runtime HW). 
\textbf{Hyper-threading (SMT):} hardware multithreading $+$ superscalar pipelining (combines TLP $+$ ILP).

\subsection{Multiprocessors (MIMD)}
\begin{outline}
  \1 \textbf{Coupling:} \emph{tightly coupled} (shared memory) vs.\ \emph{loosely coupled} (message passing over fast networks).
  \1 \textbf{Task division:} \textbf{AMP} (one master controls others, own address space per CPU) vs.\ \textbf{SMP} (every CPU can do any task, shared address space).
  \1 \textbf{UMA} (uniform access via bus) vs.\ \textbf{NUMA} (local memory per CPU, different access times).
\end{outline}

\subsubsection{RTOS on heterogeneous Multiprocessors}
% GOAL L11 • You can outline how FreeRTOS is run on a heterogeneous multiprocessor system.
% GOAL L11 • You can describe how the different synchronization primitives are implemented to work with multiple CPU cores.

On a \textbf{homogenous} multiprocessor (like the Raspi Pico), a singe RTOS instance with a shared task pool can be used.
Typical sync principles can be used like spinlocks or atomic operations.

On a \textbf{heterogeneous} multiprocessor (like the STM32 H755), each CPU has its own RTOS instance with \textbf{independent scheduling}.
Shared memory can be used to exchange data.
Hardware semaphores (\textbf{HSEM}) must be used for shared access.
Some interrupt lines are available between the two CPUs, which can be used to sync the processors (see \ref{Mutex_From_ISR}).
(Shared memory can still be used).

\subsection{Amdahl's Law (strong scaling), fixed problem size}
\begin{itemize}
    \item \textbf{Rule:} The performance increase from an improvement is limited by the proportion of time the improved part is used.
    \item Application to multi-processors: Improvement $=$ parallelization; utilization rate $=$ degree of parallelizability.
    \item Guiding principles: \emph{make common case fast}; \emph{rule of diminishing returns}; another part of the system can become the new bottleneck.
\end{itemize}

\subsubsection{Derivation of Speed-up}

\begin{minipage}[t]{0.38\linewidth}
    \vspace*{0pt}
    \textit{Notation:} 

    $A$ Workload (steps)

    $L_{\text{SISD}}$, $L_{\text{MIMD}}$ Performance (steps/time)

    $n$ number of processors

    $s$ sequential part

    $p$ parallel part with $(s+p)\cdot A = A$, 
    
    so $s+p=1$, $p=1-s$.
\end{minipage}\hfill
\begin{minipage}[t]{0.6\linewidth}
    \vspace*{0pt}
    \begin{align*}
    T_m &= T_s + T_p = \frac{s\cdot A}{L_{\text{SISD}}} + \frac{p\cdot A}{n\cdot L_{\text{SISD}}} = \frac{A}{L_{\text{SISD}}}\left(s+\frac{p}{n}\right)\\
    \overline{L_{\text{MIMD}}} &= \frac{A}{T_m} = L_{\text{SISD}}\cdot\frac{1}{s+\frac{1-s}{n}} = \frac{L_{\text{SISD}}}{s+\frac{1-s}{n}}
    \end{align*}
    \[
    \boxed{\;\text{Speedup} = S(n) = \frac{\overline{L_{\text{MIMD}}}}{L_{\text{SISD}}} = \frac{1}{s+\dfrac{1-s}{n}}\;}
    \]
\end{minipage}

% Grenzfall $n\to\infty$: $\text{Speedup}\to 1/s$ (harte obere Schranke durch den seriellen Anteil).

\example{Simple}
Serial part \qty{10}{\percent}\textrightarrow$s=0.1$, parallel part \qty{90}{\percent}\textrightarrow$p=0.9$, \textbf{n} SISD units

\begin{description}
    \item[$n=1$] $S = \frac{1}{(0.1+\frac{0.9}{1})} = 1$, efficiency = $\frac{1}{1} = \qty{100}{\percent}$
    \item[$n=4$] $S = \frac{1}{(0.1+\frac{0.9}{4})} \approx 3.08$, efficiency = $\frac{3.08}{4} = \qty{77}{\percent}$
    \item[$n=15$] $S = \frac{1}{(0.1+\frac{0.9}{15})} \approx 6.25$, efficiency = $\frac{6.25}{15} = \qty{41.7}{\percent}$
    \item[$n=\infty$] $S = \lim_{n\rightarrow\infty}\frac{1}{(0.1+\frac{0.9}{\infty})} = 10$, efficiency = $\lim_{n\rightarrow\infty}\frac{6.25}{\infty} = \qty{0}{\percent}$
\end{description}

\example{Multiple parallel parts}
Serial part \qty{9}{\percent}\textrightarrow$s=0.09$, 4-times parallel part \qty{17}{\percent}\textrightarrow$p_1=0.17$, fully parallel part rest\textrightarrow$p_2=0.74$, 32 SISD units

\[
\text{Speedup}
=
\frac{1}{s+\dfrac{p_1}{n_1}+\dfrac{p_2}{n_2}}
=
\frac{1}{0.09+\dfrac{0.17}{4}+\dfrac{0.74}{32}}
=
6.43
\rightarrow
\text{efficiency:}\
\frac{6.43}{32}
\approx
\qty{20}{\percent} 
\]

\subsection{Roofline Model}
% GOAL V12 • You can relate theoretical gains to hardware-centric constraints (memory vs. compute) according to the Roofline model.

\begin{minipage}[t]{0.44\linewidth}
    \vspace*{0pt}
    Code is \textbf{compute bound} if it has high operational intensity, \textbf{memory bound} if it has low operational intensity.
    
    Platform: \textbf{bandwidth} $\beta$ in bytes/cycle, \textbf{peak performance} $\pi$ in flops/cycle per core. 
    Algorithm: work $W(n)$ in flops, data movement $Q(n)$ in bytes.
    \[ 
    I(n)
    =
    \frac{W(n)}{Q(n)}\ \text{in}\ \frac{\text{flops}}{\text{byte}},
    \ P(n)
    =
    \frac{W(n)}{T(n)}\ \text{in} \frac{\text{flops}}{\text{cycle}} 
    \]
    The \textbf{ridge point} is at $I=\pi/\beta$: below it memory-bound, above it compute-bound. 
    SIMD/ILP/TLP raise $\pi$ (higher roof) and push the ridge right (more programs become memory bound); caching raises effective $I$ (less $Q$).
\end{minipage}\hfill

\subsection{Energy Saving via Parallelism}
Dynamic power/energy with activity $\alpha$, load capacitance $C_L$, supply $V_{dd}$, frequency $f$ (with $f\sim V_{dd}$):
\[ P \sim \alpha\,C_L\,V_{dd}^2\,f, \qquad E \sim \alpha\,C_L\,V_{dd}^2\cdot(\#\text{cycles}). \]
Lowering $V_{dd}$ lowers $f$ but $E\sim V_{dd}^2$. Using \textbf{two units at $V_{dd}/2$, $f_\text{max}/2$} (TLP) or two pipeline stages (ILP) gives the same throughput at $E_2=\tfrac14 E_1$ --- parallelism at reduced voltage/frequency saves substantial energy.
