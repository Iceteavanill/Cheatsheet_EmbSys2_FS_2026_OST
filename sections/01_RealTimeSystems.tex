\subsection{Characteristics of embedded systems}
\begin{enumerate}
    \item \textbf{Interactivity}: They react via sensor input or actor output with the environment (event driven)
    \item \textbf{Efficient}: Resources are used effectively
    \item \textbf{Reliable, save}: They (should) always work predictively as designed
    \item \textbf{Specific}: They are specific for an application
    \item \textbf{Real time}: They achieve either hard or soft real time requirements
    \item \textbf{External timing}: Typically the speed they operate on is dedicated from the outside
\end{enumerate}

\subsection{Real time systems definition}

\begin{minipage}{0.5\columnwidth}
    Real time systems are computing systems that \textbf{must} react within precise time constraints to events in the environment.
    They are characterized by deadlines.
    The key property is not maximum speed, but \textbf{predictability}.
    Even under worst-case workload conditions, timing constraints must not be violated.

    In contrast, general-purpose operating systems are used when multiple tasks must be executed simultaneously.
    Core functions are:
    \begin{itemize}
        \item Parallel processes and interprocess communication
        \item Abstract and provide driver for periphery
        \item Memory management, file systems
        \item System services (clock, math, multi user)
    \end{itemize}
\end{minipage}

\subsubsection{Advantages of staying bare metal}
\begin{itemize}
    \item Simple structure (for small applications)
    \item Low stack usage (only one stack required)
\end{itemize}

\subsubsection{Disadvantages of staying bare metal}
\begin{itemize}
    \item No “delay” capability
    \item Higher power consumption due to the lack of a power save mode in most architectures
    \item Difficult to maintain as program grows
    \item Tight coupling: timing and behavior of one part can affect the whole system
    \item Defeats modular programming
    \item Real time behavior only with interrupts
\end{itemize}

\subsubsection{Execution modes and privilege levels}
% Another important mechanism is the execution mode.
% An OS has at least 2 privilege levels / modes for executing instructions.
% For ARM  these break down to privileged and unprivileged mode.
% Privileged mode allows for execution of all instructions.
% Unprivileged mode does \textbf{not allow} execution of all instructions and is ment for unknown user applications

Another important mechanism is the execution mode.
An OS typically provides at least two privilege levels:

\begin{itemize}
    \item \textbf{Privileged mode}: full access to all instructions and system resources.
    \item \textbf{Unprivileged mode}: restricted access; used for user applications.
\end{itemize}

On ARM systems, these 2 modes are commonly used to separate kernel and user execution.

\subsection{Simple RTOS implementation}

\vspace{-0.2cm}
\begin{minipage}{0.48\columnwidth}
    \includegraphics[width=\columnwidth]{images/PDF/Scheduler.pdf}
\end{minipage}
\begin{minipage}{0.48\columnwidth}
    \begin{enumerate}
        \item When \verb|Addtask| is called, the task is added to the ready queue.
        \item The Scheduler periodically checks if tasks are ready.
        \item If a task is ready to be called (delay expired and priority sufficient), it is moved to the dispatcher queue.
        \item The scheduler calls the dispatcher queue, which executes all queued tasks.
    \end{enumerate}
\end{minipage}
