  \subsubsection{Cache / Scratchpad / Main Memory}
  % GOAL L07 • You can tell the difference between cache, scratchpad and main memory.
  \begin{itemize}
    \item Cache: Hardware-managed, transparent, fast access
    \item Scratchpad / TCM (Tightly Coupled Memory): Software-managed, deterministic access, no cache, typically not accessible by DMA
    \begin{itemize}
      \item ITCM: Instruction-only memory, used for fast code execution
      \item DTCM: Data memory, not accessible by DMA
    \end{itemize}
    \item Main Memory (SRAM/DRAM): Shared system RAM, slower than cache, accessed by CPU via cache or directly by DMA
  \end{itemize}

      \begin{itemize}
        \item \textcolor{codeGreen}{Read-hit:} Fast access form cache
        \item \textcolor{red}{Read-miss:} Load full cache line
        \item \textcolor{codeGreen}{Write-hit:} Depends on policy
        \begin{itemize}
          \item Write-through: Update cache \& memory
          \item Write-back: Update cache only (dirty bit set)
        \end{itemize}
        \item \textcolor{red}{Write-miss:} Write-allocate or no-write-allocate
      \end{itemize}
      \textbf{Typical combinations:}
      \begin{itemize}
        \item Write-back + Write-allocate\\ \textrightarrow~maximize write \textcolor{codeGreen}{hits}
        \item Write-through + No-Write-allocate\\ \textrightarrow~allocation gives no performance gain 
      \end{itemize}
    \end{minipage}

  \subsubsection{Replacement Policies}
  % GOAL V07 • You understand the cache accesses (read, write) and different cache replacement strategies.
    \begin{itemize}
      \item \textbf{LRU (Least Recently Used):} Replace least recently used line.
        Assumes entries unused for a long time are unlikely to be used again soon.
      \item \textbf{Random:} Random (or pseudo-random) eviction.
        For large caches (e.g., $n \geq 4$ ways), gives results comparable to LRU.
      \item \textbf{FIFO:} Replace oldest inserted line.
      \item \textbf{LFU (Least Frequently Used):} Replace the least frequently used line.
      \item \textbf{Pseudo-LRU:} Hardware approximation of LRU, used because true LRU is expensive to implement (e.g., 3 history bits per cache line set, as in the Intel 486 \qty{8}{\kilo\byte} 4-way set-associative L1 cache).
    \end{itemize}

  \subsubsection{Cache Performance}
  % GOAL V07 • You know how to assess the cache performance.
    Single-level cache:
    \[\boxed{r_{hit}=\frac{N_{hit}}{N_{access}}} \quad \boxed{r_{miss}= 1 - r_{hit}} \quad \boxed{T_{eff}=T_{hit}+r_{miss}\cdot A_{miss}}\]

    Two-level cache hierarchy (L1 + L2):
    \[
      \boxed{T_{eff} = T_{Hit\text{-}L1} + r_{Miss\text{-}L1} \cdot \left( T_{Hit\text{-}L2} + r_{Miss\text{-}L2} \cdot A_{Miss\text{-}L2} \right)}
      \quad
      \boxed{\begin{aligned}
        r_{Miss\text{-}L1} &= 1 - r_{Hit\text{-}L1} \\
        r_{Miss\text{-}L2} &= 1 - r_{Hit\text{-}L2}
      \end{aligned}}
    \]

    \textbf{Influencing factors:} cache size, line length, associativity ($n$), DMA block vs.\ line length, bus bandwidth, software support.

    \textbf{Limitations:}
    \begin{itemize}
      \item \textcolor{codeGreen}{Hit} only from 2\textsuperscript{nd} access onward
      \item \textcolor{red}{Read-miss} loads full line (mitigated by bursts)
      \item Writing saves no time (write-through) or needs later write-back
      \item Full cache \textrightarrow\ replacement algorithm needed (usually Pseudo-LRU)
      \item I/O space must not be cached
      \item Only main memory accesses are sped up, not registers/ALU
    \end{itemize}

  \subsubsection{Cache Coherence and Consistency}
  % GOAL V07 • You are aware of possible issues and solutions to ensure cache coherence and cache consistency.
    \textbf{Coherence:} caches \& main memory must agree on shared data; occurs in multi-CPU \emph{and} single-CPU multi-level-cache systems.

    \begin{minipage}[t]{0.6\columnwidth}
      \textbf{Issues:} 
      \begin{itemize}
        \item \textcolor{orange}{DMA writes to SRAM} \textrightarrow\ \textbf{invalidate} line
        \item \textcolor{orange}{DMA reads from SRAM} \textrightarrow\ \textbf{clean} (write-back)\\line
        \item \textcolor{orange}{CPU0--CPU1 write-through} \textrightarrow\ write propagation
        \item \textcolor{orange}{CPU0--CPU1 write-back} \textrightarrow\ write serialization
      \end{itemize}
    \end{minipage}\hfill
    \begin{minipage}[t]{0.36\columnwidth}
      \textbf{Solutions:}
      \begin{enumerate}
        \item Don't cache shared data\\(slower)
        \item All caches write-through\\+ bus snooping
        \item Coherence protocol\\(e.g.\ \textbf{MESI})
      \end{enumerate}
    \end{minipage}

    \paragraph{MESI Protocol}
    \small
    \begin{tabular}{@{}lll@{}}
      \textbf{M} & \textbf{(= Modified)}  & only here, dirty; exclusive R/W, no memory update \\
      \textbf{E} & \textbf{(= Exclusive)} & only here, clean; exclusive read; write $\rightarrow$ M \\
      \textbf{S} & \textbf{(= Shared)}    & in multiple caches, consistent; write $\rightarrow$ E, others $\rightarrow$ I \\
      \textbf{I} & \textbf{(= Invalid)}   & stale/empty; reload $\rightarrow$ S (WT) or E (WB) \\
    \end{tabular}

    \item Types:
    \begin{itemize}

  \subsubsection{Serialization and Deserialization Process}
  % GOAL L09 • You understand how data is serialized and deserialized.

    \begin{itemize}
      \item \textbf{Serialization}: Encode an in-memory data structure into a byte stream/text according to a defined schema/grammar (e.g. BNF).
      \item \textbf{Deserialization}: Parse the byte stream/text using the same schema/grammar and reconstruct the data structure. % (or, for zero-copy formats, provide accessor functions that interpret the buffer directly without full reconstruction).
      \item Serialization defines \emph{representation}; a \textbf{protocol} defines \emph{transmission} (e.g. framing, message boundaries).
      A protocol usually operates ``underneath'' the serialization layer. % (e.g. OSI layer 4 vs. layer 6).
    \end{itemize}

    % Typical embedded process:
    \begin{enumerate}
      \item Define a grammar/schema for the message format.
      \item Implement a parser (deserializer) for incoming data, with error handling for malformed input.
      \item Implement a serializer that writes outgoing data fields according to the schema.
      \item Wrap messages with a transport-layer protocol (e.g. length-prefix) if message boundaries are not otherwise delimited.
    \end{enumerate}

  \subsubsection{FlatBuffers (Embedded Usage)} % TODO shorten this!
  % GOAL L09 • You know how to use the FlatBuffers serialization library.

    \begin{itemize}
      \item Is a binary serialization library.
      \item A schema (\texttt{.fbs}) defines message \texttt{table}s, from which language-specific code is generated.
      \item Supports \textbf{zero-copy} access: fields are read directly from the buffer without full deserialization, reducing memory usage and CPU overhead.
      \item \textbf{union} allows different message types to share one interface; an implicit type tag identifies the contained message.

        % Example schema pattern:
\begin{lstlisting}
namespace CommunicationObjects;

table CommandMessage { number:int; }
table DataMessage    { timestamp:int; temperature:float; }

union Payload { CommandMessage, DataMessage }

table Message { message:Payload; }

root_type Message;\end{lstlisting}

      \item Workflow on the microcontroller: Build the FlatBuffer object (using the generated builder API) $\rightarrow$ obtain the resulting byte buffer $\rightarrow$ send it via the transport protocol. On reception: pass the byte buffer to the generated root accessor $\rightarrow$ read fields directly (zero-copy).
    \end{itemize}

  \subsubsection{Format Selection Criteria}
  % GOAL L09 • You can select an appropriate serialization format given a program’s requirements.

  Decision factors:
  \begin{itemize}
    \item \textbf{Target audience / tooling}: if non-technical users or third-party software (e.g. Excel) must consume the data directly, prefer a simple, human-readable, widely supported format like \textbf{CSV} (e.g. medical datalogger example: CSV chosen so medical staff can import data directly without custom tooling).
    \item \textbf{Structural complexity}: flat/tabular data $\rightarrow$ CSV; nested/hierarchical/dynamic data $\rightarrow$ JSON (or FlatBuffers if also resource-constrained).
    \item \textbf{Resource constraints (RAM/CPU/bandwidth)}: highly constrained embedded systems with frequent (de)serialization and tight real-time requirements $\rightarrow$ \textbf{FlatBuffers} (binary, zero-copy).
    \item \textbf{Cross-language interoperability}: if the same schema must be used by firmware and several other languages (PC tools, web apps) $\rightarrow$ FlatBuffers' schema-driven code generation is advantageous.
    \item \textbf{Debuggability during development}: text formats (CSV/JSON) are easier to inspect manually (e.g. over a terminal) than binary formats.
  \end{itemize}

  \subsubsection{BNF Syntax}
    Backus-Naur Form (BNF) is a notation for describing context-free grammars. A grammar consists of:
      \begin{itemize}
        \item \textbf{Terminals}: Literal symbols/characters that appear in the final string (e.g. \texttt{'a'}, \texttt{','}, \texttt{'\{'}).
        \item \textbf{Non-terminals}: Placeholders written as \texttt{<name>}, which are recursively expanded using production rules.
        \item \textbf{Production rule}: \texttt{<symbol> ::= expression}, read as "a \texttt{<symbol>} can be replaced by \texttt{expression}".
        \item \textbf{Alternation} \texttt{|}: separates alternative productions (logical OR).
        \item \textbf{Concatenation}: Sequence of terminals/non-terminals next to each other (logical AND / sequence).
        \item \textbf{Recursion}: A non-terminal may (directly or indirectly) refer to itself, enabling repetition of arbitrary length (e.g. lists, numbers of arbitrary digit count).
      \end{itemize}

      General template:
      \[\texttt{<A>} ::= \texttt{<B>} \;\texttt{<C>} \;|\; \texttt{<B>} \, \texttt{','} \, \texttt{<A>}\]
      This reads: an $A$ is either a $B$ followed by a $C$, \emph{or} a $B$ followed by a comma followed by another $A$ (recursive, allows arbitrarily long comma-separated lists).

% TODO: Cross check with Lecture!
    \subsubsection{Structured Data Representation}
      To check whether a data stream is \emph{deserializable} according to a BNF:
      \begin{enumerate}
        \item Start at the root non-terminal (e.g. \texttt{<message>}).
        \item Try to match the input stream against one of the alternative productions, left to right.
        \item Recursively expand non-terminals until only terminals remain, consuming input symbols accordingly.
        \item If the entire input is consumed and matches exactly one valid derivation, the stream is deserializable; otherwise it is malformed (parser should raise/signal an error).
      \end{enumerate}

      Practical parsing/serialization notes (cf. Lab 09):
      \begin{itemize}
        \item A \textbf{protocol} (e.g. length-prefixing) operates "underneath" serialization — it tells the receiver how many bytes a message consists of (OSI layer 4), while the BNF/serialization format defines how the bytes are structured and interpreted (OSI layer 6).
        \item Text-based formats (CSV, JSON) are simple/human-readable but verbose (more parsing/transmission overhead); binary formats (FlatBuffers) are compact and fast (zero-copy deserialization) but not human-readable.
        \item Common digit/number/string sub-grammars:
        \[
          \texttt{<digit>} ::= \texttt{'0'} \,|\, \texttt{'1'} \,|\, \dots \,|\, \texttt{'9'}
        \]
        \[
          \texttt{<number>} ::= \texttt{<digit>} \,|\, \texttt{<digit>}\,\texttt{<number>}
        \]
      \end{itemize}

\subsection{Virtual Memory}
% GOAL V09 • You can explain the concept of virtual memory, its prerequisites, and how/why it is used.

  \subsubsection{Concept of Virtual Memory}
  Virtual memory decouples the address space seen by a program (\emph{virtual address space}) from the physical RAM address space (\emph{physical address space}). Each process operates as if it has its own large, contiguous, private memory; a hardware Memory Management Unit (MMU) translates virtual addresses to physical addresses at run time.

  Goals/benefits:
  \begin{itemize}
    \item \textbf{Isolation}: one process cannot directly access another process's physical memory $\Rightarrow$ improved security/stability (similar in spirit to MPU-based task isolation seen in Lab 07, but virtual memory is a full address-translation mechanism, not just region-based access control).
    \item \textbf{Abstraction}: programs can be linked/compiled assuming a fixed virtual address layout, independent of where they are actually loaded in physical RAM.
    \item \textbf{Overcommitment}: total virtual memory of all processes can exceed physical RAM size (unused/inactive pages can be swapped to secondary storage, e.g. disk/flash).
    \item \textbf{Simplifies memory management}: avoids external fragmentation of physical memory, since virtual contiguous memory need not be physically contiguous.
  \end{itemize}

  \subsubsection{Use Cases and Requirements}
  Prerequisites for virtual memory:
  \begin{itemize}
    \item Hardware MMU capable of address translation (table-based, e.g. page tables) and triggering faults on invalid access (\emph{page fault}).
    \item An operating system component to manage page tables, handle page faults, and (optionally) swap pages to/from secondary storage.
    \item Sufficient secondary storage if overcommitment/swapping is desired.
  \end{itemize}
  Typical use cases: general-purpose OSes (desktop/server) running multiple untrusted processes; \emph{not} typically used on small microcontrollers (e.g. Cortex-M class, as used throughout this course), which instead rely on simpler mechanisms such as the MPU for protection without full address translation, due to cost/complexity/real-time constraints.

\subsection{Paging}
% GOAL V09 • You can understand the basics of address transformations and explain specifically how the page transformation works.

  \subsubsection{Address Translation}
    A virtual address (VA) is split into:
    \[VA = \underbrace{\text{Virtual Page Number (VPN)}}_{\text{high-order bits}} \;\|\; \underbrace{\text{Page Offset}}_{\text{low-order bits}}\]
    The number of offset bits is determined by the page size:
    \[\text{offset bits} = \log_2(\text{page size})\]
    e.g. for a 4 KiB page size: $\log_2(4096) = 12$ offset bits.

    Translation steps:
    \begin{enumerate}
      \item Extract VPN from the virtual address.
      \item Look up VPN in the page table $\rightarrow$ obtain corresponding Physical Frame Number (PFN).
      \item Concatenate PFN with the (unchanged) page offset to form the physical address:
      \[PA = PFN \;\|\; \text{Page Offset}\]
      \item If no valid mapping exists (page table entry invalid / not present), the MMU raises a \emph{page fault} exception, handled by the OS (e.g. load page from disk, or terminate process on illegal access).
    \end{enumerate}

  \subsubsection{Page Tables}
    \begin{itemize}
      \item A \textbf{page table} maps VPNs to PFNs, one entry per page (\emph{Page Table Entry}, PTE).
      \item Each PTE typically contains: PFN, valid/present bit, access permissions (read/write/execute), dirty bit, accessed bit, cacheability attributes.
      \item Single-level (flat) page tables can be very large; \textbf{multi-level (hierarchical) page tables} split the VPN into several indices, each indexing into a different table level, so unused regions of the address space need no table entries — saving memory.
      \item A \textbf{Translation Lookaside Buffer (TLB)} caches recent VPN$\to$PFN translations to avoid the cost of walking page tables on every memory access.
    \end{itemize}

  \subsubsection{Page Transformation Mechanism}
    Example (single-level page table), given:
    \begin{itemize}
      \item Virtual address width: $V$ bits
      \item Page size: $P$ bytes $\Rightarrow$ offset width $= \log_2 P$ bits
      \item Number of virtual pages: $2^{V - \log_2 P}$
    \end{itemize}
    Number of page table entries required:
    \[\#\text{PTE} = 2^{(V - \log_2 P)}\]
    Total page table size (if entry size $= E$ bytes):
    \[\text{Table Size} = \#\text{PTE} \times E\]

    Worked example pattern (fill in with given numbers in an exam question):
    \begin{enumerate}
      \item Determine page size $\Rightarrow$ offset bits.
      \item Subtract offset bits from total VA width $\Rightarrow$ VPN bits.
      \item $2^{\text{VPN bits}}$ gives the number of page table entries (and hence max. number of pages addressable).
      \item Multiply by PTE size to get total page table memory footprint.
    \end{enumerate}
