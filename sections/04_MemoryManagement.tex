  \subsubsection{Cache / Scratchpad / Main Memory}
  % GOAL L07 • You can tell the difference between cache, scratchpad and main memory.
  \begin{itemize}
    \item Cache: Hardware-managed, transparent, fast access
    \item Scratchpad / TCM (Tightly Coupled Memory): Software-managed, deterministic access, no cache, typically not accessible by DMA
    \begin{itemize}
      \item ITCM: Instruction-only memory, used for fast code execution
      \item DTCM: Data memory, not accessible by DMA
    \end{itemize}
    \item Main Memory (SRAM/DRAM): Shared system RAM, slower than cache, accessed by CPU via cache or directly by DMA
  \end{itemize}

\subsection{Memory Protection Unit (MPU)}
  The MPU is a hardware unit integrated into the CPU that controls access to memory regions.

  \begin{minipage}[t]{0.58\linewidth}
    % GOAL V06 • You can describe the purpose of access levels and a memory protection unit.
    \subsubsection{Purpose}
    \begin{itemize}
      \item Region-based memory protection
      \item Access permissions: Read / Write / Execute
      \item Privileged vs.~unprivileged access
      \item Illegal access \textrightarrow~\textcolor{red}{Memory Management Fault}
      \item Prevents memory corruption and isolates OS/tasks
    \end{itemize}

    \subsubsection{Operating Modes \& Access Levels}
    \begin{itemize}
      \item \textbf{Handler mode:} Interrupts, always Privileged
      \item \textbf{Thread mode:} Normal program execution
      \item \textbf{Privileged:} Full CPU and memory access
      \item \textbf{Unprivileged:} Restricted access according to MPU
    \end{itemize}
  \end{minipage}\hfill
  \begin{minipage}[t]{0.38\linewidth}
    \subsubsection{FreeRTOS}
    % GOAL L07 • You can employ Arm’s access levels to manage privileges of tasks and user programs in FreeRTOS.
    \begin{itemize}
      \item Kernel runs privileged
      \item User tasks run unprivileged
      \item System calls (SVC) request privileged kernel services
      \item MPU protects kernel memory from user tasks
    \end{itemize}

    \subsubsection{MPU Regions}
    % GOAL L07 • You can use the memory protection unit to control access rights and prevent memory corruption.
    \begin{itemize}
      \item Region size: power of 2
      \item Minimum size: 32 bytes
      \item Region base address aligned to its size
    \end{itemize}
  \end{minipage}

  \subsubsection{Cache Accesses}
  % GOAL V07 • You understand the cache accesses (read, write).

    \begin{minipage}[t]{0.34\columnwidth}
      \begin{enumerate}
        \item Convert address to binary
        \item Extract Tag / Index / Offset
        \item Select line (or set)
        \item Compare Tag
        \item \textcolor{codeGreen}{Hit} / \textcolor{red}{Miss}
      \end{enumerate}
    \end{minipage}\hfill
    \begin{minipage}[t]{0.62\columnwidth}
      \begin{itemize}
        \item \textcolor{codeGreen}{Read-hit:} Fast access form cache
        \item \textcolor{red}{Read-miss:} Load full cache line
        \item \textcolor{codeGreen}{Write-hit:} Depends on policy
        \begin{itemize}
          \item Write-trough: Update cache \& memory
          \item Write-back: Update cache only (dirty bit set)
        \end{itemize}
        \item \textcolor{red}{Write-miss:} Write-allocate or no-write-allocate
      \end{itemize}
    \end{minipage}
    
  \subsubsection{Replacement Policies}
  % GOAL V07 • You understand different cache replacement strategies.
    \begin{itemize}
      \item \textbf{LRU:} Replace least recently used line
      \item \textbf{FIFO:} Replace oldest inserted line
      \item \textbf{Random:} Random eviction
      \item \textbf{Pseudo-LRU:} Hardware approximation of LRU (efficient)
    \end{itemize}

  % GOAL V07 • You are aware of possible issues and solutions to ensure cache coherence and cache consistency.
      \begin{itemize}
        \item Coherence ensures consistency between caches and main memory
        \item Problem: Same memory in multiple caches / DMA inconsistency
        \item Solution:
        \begin{itemize}
          \item Invalidate cache lines
          \item Clean (write back) dirty lines
          \item Coherence protocols (e.g. MESI)

\subsection{Flash Memory}
% GOAL V08 • You know the basics of how flash memory works and how it can be used together with a file system.
  
  \begin{itemize}
    \item Non-volatile memory (data persists without power)
    \item Based on floating-gate MOSFET (FGMOS)
    \item Types:
    \begin{itemize}
      \item NOR Flash: Random access, fast read, code storage
      \item NAND Flash: Block-based access, high density, data storage (e.g. SD cards)
    \end{itemize}
    \item Key property: \textbf{erase-before-write}
    \item Erase is \textbf{block/sector-based}, not byte-based
    \item Limited write/erase cycles \textrightarrow~wear leveling required
  \end{itemize}

  \subsubsection{Flash Structure}
  % GOAL L08 • You learn about the structure of flash memory (flash sectors) through hands-on experience.
    \begin{itemize}
      \item Flash is divided into \textbf{sectors / pages / blocks}
      \item STM32H755 example:
        \begin{itemize}
          \item Sector size: \qty{128}{\kilo\byte} (internal flash)
          \item Writing \qty{8}{\kilo\byte} \textrightarrow~only \textbf{1 sector must be erased}
        \end{itemize}
      \item Sector erase is required before any write Operations
      \item File systems map logical data \textrightarrow~flash sectors
    \end{itemize}

  \subsubsection{Flash Read/Write Operations}
  % GOAL L08 • You can read from and write to flash memory.
    \begin{itemize}
      \item Read: Byte/word access (depending on flash type, NOR supports random access)
      \item Write: Only into erased memory
      \item Sequence:
        \begin{enumerate}
          \item Erase sector/block
          \item Program (write) data
          \item Verify write (optional but recommended)
        \end{enumerate}
      \item External NOR Flash (e.g. MT25QL256):
        \begin{itemize}
          \item Requires command sequence (QSPI)
          \item Read via dedicated read function (e.g. STR read)
        \end{itemize}
        \item Internal flash often accessed via HAL / driver functions
    \end{itemize}

  \subsubsection{RAM vs. Flash Lifetime}
  % GOAL L08 • You can understand the data lifetime difference between RAM and flash memory.

  \begin{minipage}[t]{0.48\linewidth}
    RAM:
    \begin{itemize}
        \item Volatile (lost on reset)
        \item Unlimited write cycles (practically)
        \item Fast access
        \item Typical use: Firmware, persistent data
      \end{itemize}
  \end{minipage}\hfill
  \begin{minipage}[t]{0.48\linewidth}
    Flash:
    \begin{itemize}
      \item Non-volatile (persistent storage)
      \item Limited erase/write cycles
      \item Slower write/erase than RAM
      \item Typical use: Runtime variables
    \end{itemize}
  \end{minipage}

  \subsubsection{Flash Sectors}
  % NOTE: Lab relevance: STM32 + File system behavior
    \begin{itemize}
      \item Sector = smallest erase unit
      \item Important constraint:
        \begin{itemize}
          \item Sector erase \textrightarrow~overwrite data
        \end{itemize}
      \item Example (STM32H755):
        \begin{itemize}
          \item Internal flash used for code + persistent variables
          \item Sector-based storage (e.g. sector 7 used for reset counter)
        \end{itemize}
      \item External flash (MT25QL256):
        \begin{itemize}
          \item Organized in blocks/sectors
          \item Used with file systems (LittleFS / FatFS)
        \end{itemize}
    \end{itemize}

\subsection{File Systems}
% GOAL L08 • You can make use of the basic functionalities of a file system.

  \subsubsection{Basic File System Operations}
    \begin{itemize}
      \item File system provides abstraction over raw storage (Flash/SD)
      \item Typical workflow:
    \end{itemize}
    \begin{enumerate}
      \item Mount file system (initialize structures, buffers)
      \item Open file (f\_open)
      \item Read / Write data (f\_read, f\_write)
      \item Close file (f\_close)
    \end{enumerate}
    
    \begin{itemize}
      \item File system handles:
        \begin{itemize}
          \item Block allocation
          \item Metadata Management
          \item Error handling
        \end{itemize}
      \item Important: Always check return codes (robust embedded systems)
    \end{itemize}

  \subsubsection{Embedded File Systems}    
    \begin{itemize}
      \item Embedded file systems are optimized for flash memory constraints:
      \begin{itemize}
        \item Limited write cycles
        \item Erase-before-write requirements
        \item Sector-based storage
      \end{itemize}
    \end{itemize}

    \textbf{Common implementations:}    
    \begin{itemize}
      \item FatFS:
      \begin{itemize}
        \item Designed for NOR flash
        \item Built-in wear leveling
        \item Power-loss resilient (save for embedded systems)
      \end{itemize}
    \end{itemize}
