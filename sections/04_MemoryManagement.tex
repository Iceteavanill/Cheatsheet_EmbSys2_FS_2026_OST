  \subsubsection{Cache / Scratchpad / Main Memory}
  % GOAL L07 • You can tell the difference between cache, scratchpad and main memory.
  \begin{itemize}
    \item Cache: Hardware-managed, transparent, fast access
    \item Scratchpad / TCM (Tightly Coupled Memory): Software-managed, deterministic access, no cache, typically not accessible by DMA
    \begin{itemize}
      \item ITCM: Instruction-only memory, used for fast code execution
      \item DTCM: Data memory, not accessible by DMA
    \end{itemize}
    \item Main Memory (SRAM/DRAM): Shared system RAM, slower than cache, accessed by CPU via cache or directly by DMA
  \end{itemize}

\subsection{Memory Protection Unit (MPU)}
  The MPU is a hardware unit integrated into the CPU that controls access to memory regions.

  \begin{minipage}[t]{0.58\linewidth}

      \item Access permissions: Read / Write / Execute
      \item Privileged vs.~unprivileged access
      \item Illegal access \textrightarrow~\textcolor{red}{Memory Management Fault}
      \item Prevents memory corruption and isolates OS/tasks

      \textbf{Cache Line}\\
      A cache line is a block of $2^{\cacheI}$ bytes copied from main memory.
    \end{minipage}\hfill
    \begin{minipage}[t]{0.34\columnwidth}
      \textbf{Cache Organization}
      \begin{itemize}
        \item Cache contains $2^{\cacheK}$ lines
        \item Cache size: $2^{\cacheK+\cacheI}$\\
        For $n$-way: $n \cdot 2^{\cacheK+\cacheI}$
        \item Entry: Tag + Valid + Data
      \end{itemize}
    \end{minipage}\hfill
    \begin{minipage}[t]{0.30\columnwidth}
      \textbf{Block Mapping}
      \vspace{-0.3cm}
      \[\begin{array}{lcl}
        \text{Block} = \lfloor \text{Address}/2^{\cacheI}\rfloor\\
        \text{Set}   = \text{Block}\bmod 2^{\cacheK}\\
        \text{Tag}   = \lfloor \text{Block}/2^{\cacheK}\rfloor

    \end{minipage}

    Fully associative: $\#\text{sets} = 1 \Rightarrow x = \cacheK$, Direct mapped: $n = 1 \Rightarrow \cacheK = s$

        \item Convert address to binary
        \item Extract Tag / \pastelbox{cyan}{Index} / \pastelbox{green}{Offset}
        \item Select line (or set)
        \item Compare Tag
        \item \textcolor{codeGreen}{Hit} / \textcolor{red}{Miss}

    \begin{minipage}[t]{0.6\columnwidth}
      \begin{itemize}
        \item \textcolor{codeGreen}{Read-hit:} Fast access form cache
        \item \textcolor{red}{Read-miss:} Load full cache line
        \item \textcolor{codeGreen}{Write-hit:} Depends on policy
        \begin{itemize}
          \item Write-through: Update cache \& memory
          \item Write-back: Update cache only (dirty bit set)
        \end{itemize}
        \item \textcolor{red}{Write-miss:} Write-allocate or no-write-allocate
      \end{itemize}
      \textbf{Typical combinations:}
      \begin{itemize}
        \item Write-back + Write-allocate\\ \textrightarrow~maximize write \textcolor{codeGreen}{hits}
        \item Write-through + No-Write-allocate\\ \textrightarrow~allocation gives no performance gain 
      \end{itemize}
    \end{minipage}

  \subsubsection{Replacement Policies}
  % GOAL V07 • You understand the cache accesses (read, write) and different cache replacement strategies.
    \begin{itemize}
      \item \textbf{LRU (Least Recently Used):} Replace least recently used line.
        Assumes entries unused for a long time are unlikely to be used again soon.
      \item \textbf{Random:} Random (or pseudo-random) eviction.
        For large caches (e.g., $n \geq 4$ ways), gives results comparable to LRU.
      \item \textbf{FIFO:} Replace oldest inserted line.
      \item \textbf{LFU (Least Frequently Used):} Replace the least frequently used line.
      \item \textbf{Pseudo-LRU:} Hardware approximation of LRU, used because true LRU is expensive to implement (e.g., 3 history bits per cache line set, as in the Intel 486 \qty{8}{\kilo\byte} 4-way set-associative L1 cache).
    \end{itemize}

  \subsubsection{Cache Performance}
  % GOAL V07 • You know how to assess the cache performance.
$T_{eff}$ = mean acces time (s or \#clocks), $T_{Hit}$ = cache access time (s or \#clocks), $r_{Miss}$ = cache miss rate (\%), $A_{miss}$ = additional cache delay on miss (s or \#clocks)

    Single-level cache:
    \[\boxed{r_{hit}=\frac{N_{hit}}{N_{access}}} \quad \boxed{r_{miss}= 1 - r_{hit}} \quad \boxed{T_{eff}=T_{hit}+r_{miss}\cdot A_{miss}}\]

    Two-level cache hierarchy (L1 + L2):
    \[
      \boxed{T_{eff} = T_{Hit\text{-}L1} + r_{Miss\text{-}L1} \cdot \left( T_{Hit\text{-}L2} + r_{Miss\text{-}L2} \cdot A_{Miss\text{-}L2} \right)}
      \quad
      \boxed{\begin{aligned}
        r_{Miss\text{-}L1} &= 1 - r_{Hit\text{-}L1} \\
        r_{Miss\text{-}L2} &= 1 - r_{Hit\text{-}L2}
      \end{aligned}}
    \]

    \textbf{Influencing factors:} cache size, line length, associativity ($n$), DMA block vs.\ line length, bus bandwidth, software support.

    \textbf{Limitations:}
    \begin{itemize}
      \item \textcolor{codeGreen}{Hit} only from 2\textsuperscript{nd} access onward
      \item \textcolor{red}{Read-miss} loads full line (mitigated by bursts)
      \item Writing saves no time (write-through) or needs later write-back
      \item Full cache \textrightarrow\ replacement algorithm needed (usually Pseudo-LRU)
      \item I/O space must not be cached
      \item Only main memory accesses are sped up, not registers/ALU
    \end{itemize}

  \subsubsection{Cache Coherence and Consistency}
  % GOAL V07 • You are aware of possible issues and solutions to ensure cache coherence and cache consistency.
    \textbf{Coherence:} caches \& main memory must agree on shared data; occurs in multi-CPU \emph{and} single-CPU multi-level-cache systems.

    \begin{minipage}[t]{0.6\columnwidth}
      \textbf{Issues:} 
      \begin{itemize}
        \item \textcolor{orange}{DMA writes to SRAM} \textrightarrow\ \textbf{invalidate} line
        \item \textcolor{orange}{DMA reads from SRAM} \textrightarrow\ \textbf{clean} (write-back)\\line
        \item \textcolor{orange}{CPU0--CPU1 write-through} \textrightarrow\ write propagation
        \item \textcolor{orange}{CPU0--CPU1 write-back} \textrightarrow\ write serialization
      \end{itemize}
    \end{minipage}\hfill
    \begin{minipage}[t]{0.36\columnwidth}
      \textbf{Solutions:}
      \begin{enumerate}
        \item Don't cache shared data\\(slower)
        \item All caches write-through\\+ bus snooping
        \item Coherence protocol\\(e.g.\ \textbf{MESI})
      \end{enumerate}
    \end{minipage}

    \paragraph{MESI Protocol}
    \small
    \begin{tabular}{@{}lll@{}}
      \textbf{M} & \textbf{(= Modified)}  & only here, dirty; exclusive R/W, no memory update \\
      \textbf{E} & \textbf{(= Exclusive)} & only here, clean; exclusive read; write $\rightarrow$ M \\
      \textbf{S} & \textbf{(= Shared)}    & in multiple caches, consistent; write $\rightarrow$ M, others $\rightarrow$ I \\
      \textbf{I} & \textbf{(= Invalid)}   & stale/empty; reload $\rightarrow$ S (WT) or E (WB) \\
    \end{tabular}

\subsection{Flash Memory} % TODO: Maybe add an image and check if content is complete or can be shortened
% GOAL V08 • You know the basics of how flash memory works and how it can be used together with a file system.

  \begin{itemize}
    \item Non-volatile memory (data persists without power)
    \item Based on floating-gate MOSFET (FGMOS)
    \item Types:
    \begin{itemize}
      \item \textbf{NOR Flash:} Random access, fast read, code storage
      \item \textbf{NAND Flash:} Block-based access, fast write, high density, data storage (e.g. SD cards)
    \end{itemize}
    \item Key property: \textbf{erase-before-write}
    \item Erase is \textbf{block/sector-based}, not byte-based
    \item Limited write/erase cycles \textrightarrow~wear leveling required
  \end{itemize}

  \subsubsection{Serialization and Deserialization Process}
  % GOAL L09 • You understand how data is serialized and deserialized.

    \begin{itemize}
      \item \textbf{\textcolor{codeGreen}{Serialization}}: Encode an in-memory data structure into a byte stream/text according to a defined schema/grammar (e.g. BNF).
      \item \textbf{\textcolor{blue}{Deserialization}}: Parse the byte stream/text using the same schema/grammar and reconstruct the data structure. % (or, for zero-copy formats, provide accessor functions that interpret the buffer directly without full reconstruction).
      \item \textcolor{codeGreen}{Serialization} defines \emph{representation}; a \textbf{protocol} defines \emph{transmission} (e.g. framing, message boundaries).
      A protocol usually operates ``underneath'' the serialization layer. % (e.g. OSI layer 4 vs. layer 6).
    \end{itemize}

    % Typical embedded process:
    \begin{enumerate}
      \item Define a grammar/schema for the message format.
      \item Implement a parser (\textcolor{blue}{deserializer}) for incoming data, with error handling for malformed input.
      \item Implement a \textcolor{codeGreen}{serializer} that writes outgoing data fields according to the schema.
      \item Wrap messages with a transport-layer protocol (e.g.\ length-prefix) if message boundaries are not otherwise delimited.
    \end{enumerate}

  \subsubsection{Serialization Formats}
    The efficiency (size, parsing speed, memory overhead) depends entirely on the chosen \emph{format}.
    \begin{center}
      \begin{tabular}{lp{3.8cm}p{3.8cm}}
        \toprule
        \textbf{Format} & \textbf{Advantage}                                                                                                                                      & \textbf{Disadvantage}                                                                                                                   \\ \midrule
        CSV             & Simple, text-based, tabular, widely supported  (e.g.\ Excel)                                                                                            & Limited to flat data; no native support for hierarchical/nested structures                                                              \\ \midrule % (e.g. directly importable into Excel)
        JSON            & Flexible text format; supports dynamic, hierarchical/nested payloads; human-readable                                                                    & Relatively high serialization/deserialization overhead due to verbose text representation; inefficient for constrained embedded systems \\ \midrule
        FlatBuffers     & Very memory-efficient, fast binary format; zero-copy deserialization (no parsing or object reconstruction needed); cross-language compatible via schema & Rather complex; poor human readability/debuggability; requires a predefined, shared schema (less flexible/dynamic)                      \\ \bottomrule
      \end{tabular}
    \end{center}

    \subsubsection{Format Selection Criteria}
    % GOAL L09 • You can select an appropriate serialization format given a program’s requirements.

    \begin{itemize}
      \item \textbf{Non-technical consumers / tabular data} $\rightarrow$ \textbf{CSV} (e.g., medical datalogger: staff imports to Excel).
      \item \textbf{Nested / dynamic / hierarchical data} $\rightarrow$ \textbf{JSON}.
      \item \textbf{Resource-constrained / high-frequency / cross-language} $\rightarrow$ \textbf{FlatBuffers} (binary, zero-copy).
      \item \textbf{Debuggability priority} $\rightarrow$ prefer text formats (CSV/JSON) over binary.
    \end{itemize}

  \subsubsection{FlatBuffers (Embedded Usage)}
  % GOAL L09 • You know how to use the FlatBuffers serialization library.

    \begin{itemize}
      \item A schema (\texttt{.fbs}) defines message \texttt{table}s, from which language-specific code is generated.
      \item \textbf{Zero-copy}: Fields are read directly from the buffer on demand — no parsing or object reconstruction, minimal RAM and CPU overhead.
      \item \textbf{union}: Multiple message types share one interface; an implicit type tag identifies the active variant.
      \item \textbf{Workflow}: Build FlatBuffer object via generated builder API $\rightarrow$ obtain byte buffer $\rightarrow$ transmit via protocol.
                              On reception: pass buffer to root accessor $\rightarrow$ read fields zero-copy.

\subsection{BNF Grammar}
% GOAL V09 • You can apply BNF grammar definitions and explain how structured data formats are defined and interpreted.

  \subsubsection{BNF Syntax}
    Backus-Naur Form (BNF) is a notation for describing context-free grammars. A grammar consists of:
      \begin{itemize}
        \item \textbf{Terminal}: Literal character/symbol (e.g. \texttt{'a'}, \texttt{','}, \texttt{'\{'}).
        \item \textbf{Non-terminal}: Placeholder \texttt{<name>}, expanded by production rules.
        \item \textbf{Production rule}: \texttt{<symbol> ::= expression}, alternatives via \texttt{|}, concatenation implicit, recursion for repetition.
      \end{itemize}

      % General template:
      % \[\texttt{<A>} ::= \texttt{<B>}\;\texttt{<C>} \;\big|\; \texttt{<B>}\,\texttt{','}\,\texttt{<A>}\]
      % Common sub-grammars:
      \[
        \texttt{<digit>} ::= \texttt{'0'} \mid \cdots \mid \texttt{'9'} \qquad
        \texttt{<number>} ::= \texttt{<digit>} \mid \texttt{<digit>}\,\texttt{<number>}
      \]

    \subsubsection{Structured Data Representation}
      To check whether a data stream is \emph{\textcolor{blue}{deserializable}} according to a BNF:
      \begin{enumerate}
        \item Start at the root non-terminal (e.g.\ \texttt{<message>}).
        \item Match input left-to-right, recursively expanding non-terminals.
        \item Valid full match $\rightarrow$ \textcolor{blue}{deserializable}; otherwise $\rightarrow$ \textcolor{red}{malformed (signal error)}.
      \end{enumerate}
      
      Practical parsing/serialization notes:
      \begin{itemize}
        \item Characters not listed as terminals in the grammar (e.g.\ a space, or the symbol \texttt{'p'} when only \texttt{'a'}\,\texttt{'b'}\,\texttt{'c'} are defined) make the stream \textcolor{red}{non-deserializable}.
        \item Mixing terminal classes in one token that the grammar keeps separate (e.g.\ \texttt{b28} mixing symbols and digits) is \textcolor{red}{non-deserializable} even if each character individually appears in the grammar.
      \end{itemize}

\subsection{Virtual Memory}
% GOAL V09 • You can explain the concept of virtual memory, its prerequisites, and how/why it is used.

  \subsubsection{Concept}
  \begin{center}
    \includegraphics[width=0.8\linewidth]{images/PDF/VirtualMemory_Concept.pdf}
  \end{center}
  \vspace{-0.25cm}

  % Decouples the \emph{virtual address space} (seen by a program) from the \emph{physical address space} (RAM).\\
  % A hardware \textbf{MMU} translates virtual $\rightarrow$ physical addresses at run time.

  \begin{itemize}
    \item \textbf{Isolation}: Processes cannot access each other's physical memory.
    \item \textbf{Abstraction}: Programs compile against a fixed virtual layout, independent of physical placement.
    \item \textbf{Overcommitment}: Total virtual memory can exceed physical RAM (inactive pages swapped to secondary storage).
    \item \textbf{No external fragmentation}: Virtually contiguous memory need not be physically contiguous.
  \end{itemize}

  \subsubsection{Prerequisites \& Use Cases}
  \begin{itemize}
    \item Requires MMU (address translation, page faults) and OS support (page tables, fault handling, optional swapping).
    \item Used in general-purpose OSes (desktop/server) with multiple processes.
    \item Not used in Cortex-M MCUs; instead MPU-based protection is used (simpler, no address translation, lower cost/complexity).
  \end{itemize}

\subsection{Paging}
% GOAL V09 • You can understand the basics of address transformations and explain specifically how the page transformation works.

  \subsubsection{Address Translation}
    \begin{center}
      \includegraphics[width=\linewidth]{images/PDF/Paging_AddressTranslation.pdf}
    \end{center}

  \subsubsection{Page Tables}
    Map VPN $\rightarrow$ PFN via \textbf{Page Table Entries (PTEs)}.
    \textbf{PTE} fields: PFN, valid/present bit, access permissions (read/write/execute), dirty/accessed bit, cacheability.
    \begin{itemize}
      \item \textbf{Multi-level:} Allocate only needed levels $\rightarrow$ less memory.
      \item \textbf{Translation Lookaside Buffer (TLB):} Caches VPN $\rightarrow$ PFN.\\
      \textcolor{codeGreen}{TLB hit:} direct translation, \textcolor{red}{TLB miss:} page-table walk + update.
    \end{itemize}

    \textcolor{red}{Invalid page table entry} $\rightarrow$ MMU raises \emph{page fault} $\rightarrow$ OS handles fault

  
  \subsubsection{Useful Relations}
    For a virtual address width $V$, page size $P=2^k$ bytes and page table entry size $E$:

    \[
    \begin{aligned}
    \text{Offset bits}      &= \log_2(P) = k \\
    \text{VPN bits}         &= V - \log_2(P) \\
    \#\text{Virtual Pages}  &= \#\text{PTE} = 2^{\,V-\log_2(P)} \\
    \text{Page Table Size}  &= \#\text{PTE} \times E
    \end{aligned}
    \]

    \textcolor{gray}{Example: $P=\qty{4}{KiB}=2^{12}$ bytes $\Rightarrow$ Offset $=12$ bits.}

  \subsubsection{Segmentation vs.\ Paging}
    \begin{itemize}
        \item \textbf{Paging}: fixed-size pages, no external fragmentation.
        \item \textbf{Segmentation}: variable-size regions (code/data/stack), may cause external fragmentation.
    \end{itemize}
